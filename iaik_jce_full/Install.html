How to install IAIK-JCE
When reading this file you should already have unpacked the IAIK-JCE Distribution. Download versions are available from
http://jce.iaik.tugraz.at/download/.

Your installation directory will not only contain the information files you are currently reading, but also the follwoing sub-folders:

  • demo
    contains source and Javadoc of the IAIK-JCE demos, and batch and shell scripts for running some demos (LDAP search and OCSP query) on Windows or Linux, respectively
  • docs
    contains the API documentation in form of JavaDoc
  • lib
    contains the JCE library files (iaik_jce.jar, iaik_javax_crypto.jar, iaik_jce_demo.jar)
  • lib-signed
    contains the signed versions of the library files (iaik_jce.jar, iaik_jce_native_aes.jar)
  • lib-signed/trusted_library
    contains signed versions of the library files with the "Trusted Library = true" manifest entry
  • images
    contains logos and images used in the documentation

First, ensure that you have installed a Java runtime environment. This version toolkit has been successfully tested with Java runtime environments of the 1.2, 1.3, 1.4, 1.5 and 1.6 series, and also with the most recent versions of JDK 1.7 (JDK 1.5, 1.6, 1.7 are also called JDK 5, 6, 7, respectively).
Depending on your Java version, you may have to include different jar files into your classpath. If you use Java 1.4 or later, you may need to install the Java[TM] Cryptography Extension (JCE) Unlimited Strength Jurisdiction Policy Files.

Notice: The lib/iaik_jce.jar file and the lib-signed/iaik_jce.jar file are two different library files with different content!

Here is some background information.
With version 4.0 we have stopped support for Java 1.1. Please use an IAIK-JCE version <= 3.181 if you still have to use Java 1.1.
The lib/iaik_javax_crypto.jar file contains a JCE framework implemented by IAIK. This implementation does not require JCE providers to be signed. The JCE framework consists of all packages with a name starting with javax.crypto.
Despite the core of the IAIK crypto library (algorithms, encoding/decoding, certificate handling, ...), the lib/iaik_jce.jar file contains all contents of the lib/iaik_javax_crypto.jar file. In consequence, it does not make sense to include both jar files (lib/iaik_jce.jar and lib/iaik_javax_crypto.jar) into the classpath. However, this holds only for the unsigned version. The signed version of the IAIK crypto library -the lib-signed/iaik_jce.jar file- contains only the core crypto classes but not the contents of the lib/iaik_javax_crypto.jar file (the JCE framework classes).
For Java 1.4 or later, you only need to include the signed version of the IAIK crypto library, the lib-signed/iaik_jce.jar file, because Java versions 1.4 and later already contain a JCE framework. This included version of the JCE framework (compatible to version JCE 1.2.2) requires JCE providers to be signed. Please notice that there are some versions of JDK 1.3 which include a JCE framework (version JCE 1.2.1 or later) which also requires a signed crypto provider. For instance, many IBM Java 1.3 versions require that you use the signed provider jar file; i.e. you must treat them as if they were Java 1.4. Remember, if you use Java 1.4, don't forget to install the unlimited strength jurisdiction policy files or use the unlimited strength version of IAIK-JCE.

Notice: Because of license issues regarding the use of the RC5 and RC6 algorithms, the US distribution does not contain any RC5 or RC6 implementation. However, the standard JCE version can be easily extended to support RC5 and RC6 by downloading and installing the iaik_rc.jar RC5/RC6 extension library file.
Because the IDEA algorithm is also subject to specific patent regulations, the standard version does not contain IDEA either. The iaik_idea.jar IDEA extension may be downloaded and installed to extend the standard version.
Also the use of Ephemeral Static Diffie Hellman (ESDH) may be subject to patent regulations; so ESDH is also not included in the standard distribution. The iaik_esdh.jar extension library file may be downloaded and installed for using ESDH.
All patent related jar files above (iaik_rc.jar, iaik_idea.jar and iaik_esdh.jar) are contained in one single zip archive file (iaik_jce_patented.zip) to can downloaded from our website if required.
If you want to use all available algorithms, you may alternatively download the iaik_jce_full.zip. This zip-file contains a version of the IAIK-JCE that includes all patented algorithms (i.e. IDEA, RC5/6, and ESDH).

Summing up, the following library files are available:

iaik_jce.jar This is the default version included in the distribution. This version may or may not contain classes for RC5, RC6, IDEA, and ESDH, depending on the zip-file you have obtained.
iaik_rc.jar This Java Archive File contains ONLY the classes for RC5, RC6
iaik_idea.jar This Java Archive File contains ONLY the classes for IDEA
iaik_esdh.jar This Java Archive File contains ONLY the classes for ESDH
iaik_javax_crypto.jar This Java Archive File contains the JCE framework classes implemented by IAIK.
iaik_jce_demo.jar This Java Archive File contains the demo classes

Please note that (except for iaik_jce_demo.jar and iaik_javax_crypto.jar) any of the archive files listed in the table above includes classes implementing some JCE engine. For that reason each of these jar files is available in two versions: unsigned or signed, the latter to be used with a JCE framework (like the one included in JDK1.4 or later) that requires a JCE provider to sign its JCE archive files. The signed versions of iaik_jce.jar do not contain our JCE framework implementation (javax.crypto package) because it is not required when used with an alternative JCE framework.

Classpath Example

Once you have installed IAIK-JCE, and (optionally) downloaded any extension library files, you have to properly update your CLASSPATH or your development environment settings.

On Windows, you can use the set classpath command:

> set CLASSPATH=%classpath%;<your-install-directory>\lib\iaik_jce.jar;

If you use some patented/licensed algorithm (e.g. IDEA) you also have to include the corresponding library file into the classpath (or use iaik_jce.jar from the full zip-file immediately instead of the version of the us distribution), e.g.:

> set CLASSPATH=%classpath%;<your-install-directory>\lib\iaik_idea.jar;

For running the demos include iaik_jce_demo.jar into the classpath, e.g.:

> set CLASSPATH=%classpath%;<your-install-directory>\lib\iaik_jce_demo.jar;

Please assure to take the proper iaik_jce.jar file depending on the JCE framework you are using. As noted above, the unsigned version of iaik_jce.jar contains our JCE framework implementation (JAVAX CRYPTO) and therefore maybe used with any JDK that does not include the JCE framework. When using JDK1.4 or later you will have to use the signed version of iaik_jce.jar.

LdapURLConnection

Class LdapURLConnection of package iaik.x509.net.ldap internally uses the Java Naming and Directory InterfaceTM (JNDI); thus when using LdapURLConnection you must have the required JNDI jar files in your classpath. If you are using JDK 1.3 or later you do not have to take care about it since JNDI is already contained in the standard JDK API. When using a JDK version < 1.3, you can get the required packages (jndi.jar, ldap.jar, providerutil.jar) from the JNDI homepage at SUN/ORACLE.

Unlimited Strengh Cryptography

Due to import restrictions of some countries JDK versions >= 1.4 contain so-called jurisdiction policy files only allowing strong but not unlimited cryptography. Trying to use Cipher keys that exceed the restricted key size limit may cause errors mentioning unsupported key sizes. However, since most countries do not have any crypto restrictions, unlimited versions of the jurisdiction policy files are available to may be installed for allowing unlimited key sizes and algorithm parameters. Usually, you can download them from the same web page as the Java runtime or JDK. Usually these files have to be put into the lib/security subdirectory of your JRE; however some VMs (e.g. IBM) may require to put them into another directory (e.g. lib/ext). Please read the installation instructions that come with these policy files. Take care to install these policy files in the correct JRE installation (Note that many JDKs install two JREs by default and that the one which is used by default is not(!) the one embedded in the JDK directory. To see which one is used, you may type java −verbose −version).

IAIK-JCE (version 3.17 or later) provides special versions of iaik_jce.jar allowing to use unlimited strength cryptography also if only the default jurisdiction policy files are installed. These special versions of iaik_jce.jar are contained in the lib-signed/unltd sub-directory of the IAIK-JCE distribution. If your application does not directly create a Cipher object (if you are, e.g., accessing a KeyStore or doing document encryption with PKCS#7 or IAIK-CMS (version 4.02 or later), or, for instance, using iSaSiLk (version 4.2 or later) for SSL/TLS connections), you simply have to put this unlimited strength version of iaik_jce.jar into your classpath. If you yourself create a Cipher object you should call:


  Cipher cipher = iaik.utils.IaikSecurity.getCipher(...);
instead of calling 

  Cipher cipher = Cipher.getInstance(...);
Of course, these unlimited strength versions shall be used only in countries which do not have any restrictions to cryptographic key sizes or algorithm parameters (which are most countries). Therefore these file are not included in the IAIK-JCE install file; but can be separatly downloaded from our Website. These versions of iaik_jce.jar may be useful when you want to deploy your application in countries with no crypto restrictions, but do not want that all your end users have to manually install the unlimited strength jurisdiction policy files.

AES Addon

Some modern amd64/EMT64 CPUs include Intel's new AES-NI instruction set extension, which allows great performance gains for AES implementations. Since IAIK-JCE 5.0 we have been delivering an addon that allows the user to take advantage of AES-NI. In case your CPU supports AES-NI, include the iaik_jce_native_aes.jar into your classpath, extract the *.so and *.dll from the jar's root directory to a separate (sub-)directory and set the java.library.path to this directory. This is usually achieved by adding -Djava.library.path=(natives_directory) to the java command line.

Trusted Library

When using signed/priviledged and unsigned/sandbox components together ("mixed code") modern JDKs may require to include the "Trusted-Library=true" attribute into the manifest file of all signed components, otherwise a security warning may be raised. For that reason we provide versions of all signed IAIK-JCE jar files (see folder lib-signed/trusted_library) that may be used especially for JavaTM Java WebStart appliactions, applets or when using JavaScript code that calls code within a privileged applet.
